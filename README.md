# Aiti Guru - Тестовое задание (Frontend React)

## Техническое задание

Основной фронтенд проект команды — административная система управления, насыщенная таблицами, инструментами для редактирования и сложными формами. Данное тестовое задание максимально приближено к реальным ежедневным задачам.

### Условия
- **React** (версия 18+)
- **TypeScript** (строгая типизация обязательна)
- Работоспособность в актуальной версии Google Chrome

### Источник данных
- **API товаров:** [DummyJSON Products](https://dummyjson.com/docs/products)
- **Авторизация:** [DummyJSON Auth](https://dummyjson.com/docs/auth)

### Функциональные требования

#### Форма входа
- Валидация полей (обязательность заполнения)
- Обработка ошибок: если API возвращает ошибку, выводить уведомление или текст ошибки под полями
- Логика запоминания данных для входа:
  - Если чекбокс установлен — токен сохраняется так, что сессия живёт после закрытия браузера
  - Если не установлен — сессия сбрасывается при закрытии вкладки

#### Список товаров
- Соответствие столбцам из макета Figma
- Прогресс-бар при подгрузке
- Подгрузка данных из API

#### Сортировка
- Возможность сортировки по столбцам (например, по цене или рейтингу)
- Сохранение состояния сортировки

#### Добавление товара
- По нажатию кнопки "Добавить" открывается форма с полями: Наименование, цена, вендор, артикул
- При успешном добавлении показывается Toast уведомление
- Логику сохранения через API делать не нужно

#### Логика интерфейса
- Если рейтинг товара ниже 3, значение подсвечивается красным цветом

#### Поиск товаров
- Использовать API

## Реализация

### Стек технологий

#### Основные технологии
- **React 18** — библиотека для построения UI
- **TypeScript** — строгая типизация
- **Vite** — сборщик проекта
- **SCSS Modules** — стилизация компонентов

#### State Management
- **Zustand** — легковесный state manager
  - Выбран за простоту API, минимальный boilerplate
  - Отличная производительность без лишних ре-рендеров
  - Не требует провайдеров и обёрток

#### Формы и валидация
- **React Hook Form** — управление формами
  - Высокая производительность (uncontrolled components)
  - Встроенная валидация
  - Минимальное количество ре-рендеров

#### HTTP клиент
- **Axios** — для работы с API
  - Удобные интерцепторы для токенов
  - Автоматическая обработка JSON

#### Роутинг
- **React Router** — навигация между страницами

### Архитектура

Проект построен по методологии **Feature-Sliced Design (FSD)**:

```
src/
├── app/              # Инициализация приложения
│   ├── providers/    # Роутер
│   └── styles/       # Глобальные стили
├── pages/            # Страницы приложения
│   ├── login/        # Страница авторизации
│   └── products/     # Страница товаров
├── widgets/          # Сложные составные компоненты
│   └── product-table/
├── features/         # Бизнес-функции
│   ├── auth/         # Авторизация
│   └── product/      # Работа с товарами
├── entities/         # Бизнес-сущности
│   ├── auth/         # Сущность пользователя
│   └── product/      # Сущность товара
└── shared/           # Переиспользуемый код
    ├── api/          # API клиент
    └── ui/           # UI компоненты
```

## Git Flow

### Стратегия ветвления

В проекте используется упрощённый Git Flow с одной основной веткой `master`:

```
master (main) — стабильная версия проекта
```

### Типы коммитов

Используется [Conventional Commits](https://www.conventionalcommits.org/) для структурирования истории:

| Тип | Описание | Пример |
|-----|----------|--------|
| `feat` | Новая функциональность | `feat: добавление формы авторизации` |
| `fix` | Исправление багов | `fix: исправление валидации email` |
| `style` | Стилизация, форматирование | `style: стилизация компонента Button` |
| `refactor` | Рефакторинг без изменения функционала | `refactor: упрощение логики store` |
| `chore` | Настройка проекта, зависимости | `chore: добавление ESLint конфигурации` |
| `docs` | Документация | `docs: обновление README` |
| `test` | Тесты | `test: добавление тестов для LoginForm` |

### Структура коммитов

Каждый коммит должен содержать:
1. **Заголовок** — краткое описание (не более 50 символов)
2. **Тело** — детальное описание изменений (опционально)

```
<тип>: <краткое описание>

- Детальное изменение 1
- Детальное изменение 2
```

### Пример истории коммитов

```
e19a8f7 chore: добавление package.json
424aaa5 chore: добавление package-lock.json
2685da5 chore: настройка TypeScript
23498d0 chore: настройка Vite
2a1f546 chore: добавление ESLint
f59e586 chore: добавление точки входа
b84b83f style: добавление SCSS переменных
20b7631 style: добавление глобальных стилей
3873382 feat: создание точки входа приложения
ffaa322 feat: создание корневого компонента App
...
```

### Правила именования веток

Для будущей командной работы:
- `feature/название-фичи` — новая функциональность
- `fix/название-исправления` — исправление багов
- `refactor/название` — рефакторинг

### Pull Request процесс

1. Создать ветку от `master`
2. Сделать изменения с атомарными коммитами
3. Открыть Pull Request
4. Пройти code review
5. Слить в `master` с squash (опционально)
